DEC-006: Graph Database Selection. Let me think through the options carefully in the context of this project.

**Project context:**
- CLI tool for repository integrity validation
- Current users: single user (author), potentially other DSM spoke projects
- Current graph: NetworkX DiGraph, ~30 files, ~500 sections, builds in 104ms, 12.7MB memory
- Key graph operations: most-referenced sections, orphan sections, reference chains, export to GraphML
- Epoch 3 goals: persistent graph storage, temporal queries (git-ref), cross-repo edges, entity inventory

**Option A: Full Neo4j**
Pros:
- Industry standard, massive ecosystem, Cypher is well-documented
- Web browser UI (Neo4j Browser) for visual exploration
- Rich query language for complex traversals
- Graph Data Science library for algorithms
- Clear path to production/multi-user scenarios
Cons:
- Requires Docker or native install (heavy for a CLI tool)
- Users must start Neo4j before using graph features
- CI requires service containers
- Overkill for current scale (~500 nodes)
- Adds operational complexity to what is currently a pip-install-and-run tool

**Option B: FalkorDBLite**
Pros:
- Embedded (subprocess), zero-config, pip install
- Cypher-compatible (same query language skills)
- No Docker needed
- Fits the CLI tool model (pip install, run, done)
- Good for local prototyping and testing
Cons:
- Younger project, smaller community
- Successor to RedisGraph (EOL Jan 2025), inherits some uncertainty
- Less mature ecosystem (no Graph Data Science equivalent)
- May not scale to larger repositories (though current scale is small)
- Limited web visualization options

**Option C: Dual backend**
Pros:
- Maximum flexibility, user chooses based on needs
- Abstract the graph layer, future-proof
Cons:
- Significant engineering overhead (abstraction layer)
- Lowest common denominator API limits features
- More code to maintain and test
- Premature abstraction for current needs

**Option D: Keep NetworkX + persistence**
Pros:
- No new dependency at all
- Serialize to JSON/pickle, reload on demand
- Already working, proven performance
- Simplest option
Cons:
- No persistent query engine
- No Cypher (custom Python query functions only)
- Limited to in-memory analysis
- Doesn't advance toward the graph database exploration goal

**My recommendation thinking:**

The project is called "DSM Graph Explorer." The long-term vision includes Neo4j. But the current scale doesn't need it. The key question is: what does Epoch 3 actually need?

Epoch 3 goals:
1. Persistent graph storage — any option except D provides this
2. Temporal queries (git-ref) — this is about compiling multiple graph snapshots, not about the DB engine
3. Cross-repo edges — this is about data model, not engine
4. Entity inventory — this is about format design, not engine

Actually, the temporal/cross-repo/entity work is mostly about the compilation pipeline, not the database backend. The "Neo4j migration" might be conflating two concerns: (a) making the graph persistent/queryable and (b) adding new graph features.

A pragmatic path: start with FalkorDBLite for embedded graph persistence (zero friction), build the temporal/cross-repo features, then migrate to full Neo4j when the scale or multi-user needs justify it. This avoids the Docker dependency in Epoch 3 while still delivering persistent Cypher queries.

Or even more pragmatic: Option D (NetworkX + JSON persistence) for Epoch 3, focused on the temporal/entity/cross-repo features, deferring the database migration to Epoch 4 when there's a real need.
