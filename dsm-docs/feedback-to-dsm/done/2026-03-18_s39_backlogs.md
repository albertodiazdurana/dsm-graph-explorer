## Proposal #43: Mandatory Experiment Gate for Capability-Introducing Sprints

**Date:** 2026-03-18
**Session:** 39
**Source:** Entry 48 (experiment coverage gap)
**Targets:** DSM 2.0 (PM Guidelines, Sprint Planning Template), DSM 4.0 (Section 4)

**Problem:** Epoch plans may define experiments for some sprints but not others. There is no enforcement mechanism to ensure every capability-introducing sprint has an experiment. Epochs 2-3 maintained this pattern informally; Epoch 4 dropped it.

**Proposal:** Add an "Experiment Gate" checklist item to the sprint planning template (DSM 2.0 Template 8):

> **Before sprint implementation begins:**
> - [ ] If sprint introduces new user-facing capability: EXP-XXX defined in epoch plan
> - [ ] If sprint is performance-only (no new capability): experiment skip justified in sprint notes

This makes the experiment requirement explicit and auditable. Performance-focused sprints (like incremental updates) may skip the experiment with justification, but the skip itself must be documented.

**Additionally:** When work emerges between sprints (e.g., heading refs from EXP-007 findings), the epoch plan should be updated with an experiment definition before the capability is considered complete.

**Expected benefit:** Eliminates the gap where features ship with unit tests but no real-data validation. Maintains the experiment-per-capability pattern that proved effective in Epochs 2-3.

## Proposal #44: Sprint and Feature Branching Strategy

**Date:** 2026-03-18
**Session:** 39
**Source:** Entry 49 (branching strategy missing)
**Targets:** DSM 4.0 (Section 3, Development Protocol), DSM 2.0 (PM Guidelines)

**Problem:** All work lands directly on `master` with no branching. This limits reversibility, blocks safe parallel work, and provides no isolation for experimental or incomplete changes.

**Proposal:** Adopt a branching strategy with two levels of granularity:

1. **Sprint branch (minimum):** Each sprint works on a dedicated branch (e.g., `sprint-15/protocol-usage`). Merged to `master` at sprint boundary after tests pass and sprint checklist is complete.

2. **Feature branches (for complex sprints):** Within a sprint, individual features or files can have their own branches (e.g., `sprint-15/spoke-scanner`, `sprint-15/usage-report`). Merged to the sprint branch as each feature is complete.

The choice between one sprint branch or multiple feature branches depends on sprint complexity:
- Simple sprint (1-2 files): single sprint branch is sufficient
- Complex sprint (3+ new modules): feature branches per module

**Branch naming convention:** `sprint-N/short-description` for sprint branches, same pattern for feature branches within a sprint. Examples:
- `sprint-15/protocol-usage` (sprint branch)
- `sprint-15/spoke-scanner` (feature branch within sprint)
- `sprint-15/usage-report` (feature branch within sprint)

Consistent, sortable by sprint number, and self-documenting from the name alone.

**Merge strategy:** Merge commits (not squash) to preserve individual commit history. Sprint branch merged to `master` with a merge commit at sprint boundary.

**Checkpoint integration:** Session baselines and lightweight checkpoints should record the current branch name, so continuations know which branch to resume on. Currently `session-baseline.txt` tracks HEAD commit and working tree status but not the branch. Adding a `# Branch` field ensures the lightweight session chain works across branches.

**Expected benefit:** Reversibility (revert one merge), collaboration safety (parallel sessions on separate branches), work-in-progress isolation (incomplete features don't pollute `master`).

## Proposal #45: Experiment Operationalization Checklist

**Date:** 2026-03-18
**Session:** 39
**Source:** Entries 51-52 (experiment framework not operationalized, reproducibility not default)
**Targets:** DSM_0.2 (agent protocol), DSM 4.0 (Section 4.4), DSM 1.0 Appendix C.1.6

**Problem:** DSM Appendix C.1.3-C.1.6 defines a comprehensive experiment framework (7-element template, folder conventions, EXPERIMENTS_REGISTRY.md, Limitation Discovery Protocol), but this project has never followed it across 8 experiments. The framework is documented but not triggered by any agent protocol.

**Proposal:** Add an "Experiment Protocol" to DSM_0.2 that the agent follows whenever an experiment begins:

> **Before running any experiment:**
> 1. [ ] Consult Appendix C.1.3 for the 7-element framework
> 2. [ ] Create folder: `data/experiments/EXP-{NNN}-{short-name}/`
> 3. [ ] Write executable script: `exp_{NNN}_{name}.py` (reproducibility is mandatory, not optional)
> 4. [ ] Define success criteria in the script or design.md BEFORE running
> 5. [ ] Run the script; capture output to results files
>
> **After running:**
> 6. [ ] Write results.md using the 7-element structure
> 7. [ ] Apply Limitation Discovery Protocol (C.1.5) if limitations found
> 8. [ ] Update EXPERIMENTS_REGISTRY.md
> 9. [ ] Assess post-experiment contribution (4.4.2)

**Additionally:** Create `data/experiments/EXPERIMENTS_REGISTRY.md` as a required project artifact (columns: ID, Name, Sprint, Date, Result, Folder). Initialize it retroactively for EXP-003 through EXP-008.

**Discoverability fix (from Entry 53):** The root cause of non-compliance is that C.1 templates are architecturally invisible, they live in the Appendices file which is never auto-loaded. The Experiment Protocol above must be added to DSM_0.2 core (not a module) as a behavioral trigger, not a passive reference. The current mention at DSM_0.2 line 345 is insufficient because it reads as documentation, not as a protocol step.

**Expected benefit:** Experiments become as rigorous and reproducible as tests. The agent consults the framework automatically rather than improvising ad-hoc execution.

## Proposal #46: Heading Parsability Convention

**Date:** 2026-03-18
**Session:** 39
**Source:** Entry 54 (design for parsability), EXP-008 findings
**Targets:** DSM_0.2 (conventions), DSM 1.0 (Section 3.3 or new appendix)

**Problem:** EXP-008 showed ~27% false positive rate in heading reference detection because generic short headings match common prose. The DSM team authors its own documents, so conventions can be prescribed to eliminate ambiguity at the source.

**Proposal:** Add three heading conventions to DSM documentation standards:

**Convention 1 (mandatory):** Any referenceable heading, at any level, MUST contain at least 4 non-stopword tokens.

Examples:
- "When to Use" (2 non-stopwords) -> "When to Use This Protocol" (4 non-stopwords)
- "Overview" (1 non-stopword) -> "Project Overview and Context" (4 non-stopwords)
- "Backlog" (1) -> "Project Backlog Management Process" (4)

**Convention 2 (recommended):** Headings used as cross-referenceable concepts SHOULD be unique across the document set.

**Convention 3 (recommended):** Protocol-level headings SHOULD include the protocol or concept name for self-description. "Session Transcript Protocol" is preferred over "Transcript Protocol" or just "Protocol."

**Enforcement:** GE linter can add a new check (e.g., W004: heading has fewer than 4 non-stopword tokens) to flag non-conforming headings. The `--heading-refs` parser uses the same threshold to filter known_headings.

**Expected benefit:** Eliminates FP at the source rather than building complex NLP filters. Improves human readability simultaneously. Aligns with the "Take a Bite" philosophy: since we author the docs, we control the format.
