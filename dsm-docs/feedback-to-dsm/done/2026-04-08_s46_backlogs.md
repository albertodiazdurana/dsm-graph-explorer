## Proposal #50: /dsm-align must produce a persistent, informative alignment report

**Source:** Session 46 observation while running `/dsm-align`
**Priority:** High
**Scope:** DSM Central `scripts/commands/dsm-align.md` (skill definition)

### Problem statement

After running `/dsm-align`, there is **no persistent record** of what the skill checked, what it found, what it fixed, or what remains open. The only artifact written to disk is `.claude/last-align.txt`, which contains four lines:

```
date: 2026-04-08
dsm-version: v1.4.9
result: warnings
warnings: 2
critical: 0
```

This is a **status marker**, not a report. It tells a future reader (human or agent) *that* two warnings existed, but not *which* two. Consequences observed in Session 46:

1. **User could not audit prior run.** When the user asked "what are the two warnings about?" hours after the run, the agent had to re-execute checks and *infer* the original warnings. There is no ground truth to check against.
2. **Agent produced a degraded report.** On the second invocation in the same day, the agent short-circuited to a brief summary instead of running all 13 steps, because the marker said "already aligned today." Step 12 of the skill mandates a full report on *every* run, but there is no mechanism enforcing it and no durable record proving it happened.
3. **Warnings silently drift across sessions.** Because warnings are not persisted, long-lived issues (e.g. a stale path in CLAUDE.md that no one has fixed for weeks) look identical to brand-new warnings on each run. There is no continuity.
4. **Inbox notification is conditional and lossy.** Step 12b writes an inbox notification *only on post-change runs*, and only lists Created/Fixed items, never warnings or collisions. Check-only runs leave zero trace.

### Root cause

The skill conflates three different outputs:

| Output | Purpose | Audience | Persistence |
|--------|---------|----------|-------------|
| Conversation report (step 12) | Immediate feedback | User in session | **Ephemeral** (scrolls away) |
| Inbox notification (step 12b) | Async handoff | Future session | Post-change only, Created/Fixed only |
| `last-align.txt` (step 13) | Staleness detection for `/dsm-go` | `/dsm-go` | Counts only, no content |

None of these three is a durable, full report. The conversation report is the most informative but disappears. The inbox notification drops warnings. The marker drops everything except counts. There is no single artifact a user or agent can open tomorrow and read to answer "what did the last alignment find?"

### Proposed solution

Add a fourth output: **a persistent, full alignment report** written on every run (both check-only and post-change).

**Location options** (pick one):
- `dsm-docs/checkpoints/alignment/YYYY-MM-DD_align-report.md` (archived, one per run)
- `.claude/last-align-report.md` (single file, overwritten each run, gitignored like `last-align.txt`)
- `dsm-docs/alignment/YYYY-MM-DD.md` (new canonical folder)

**Recommendation:** `.claude/last-align-report.md` (single file, overwritten, gitignored). Rationale: the *last* report is what matters for auditability; history lives in git if the user commits it. Matches the existing `last-align.txt` pattern.

**Contents:** the full step-12 report verbatim, plus:
- Timestamp (not just date)
- DSM version
- Explicit warning text for each warning (not just a count)
- Explicit collision text for each collision
- List of "already correct" items (so the user knows what *was* checked)
- Steps skipped and why (e.g. "Step 11 skipped: not DSM Central")

**Step 12 modification:** after printing the report to conversation, write the same content to the persistent file. Step 13 remains unchanged (the counts marker is still useful for `/dsm-go` staleness detection, it just stops being the *only* record).

**Step 12b modification:** the inbox notification should reference the persistent report file rather than duplicating its contents, and should fire on check-only runs too when warnings or collisions exist (not only on post-change runs).

### Acceptance criteria

1. After any `/dsm-align` run, `.claude/last-align-report.md` exists and contains the full step-12 report with explicit warning/collision text.
2. A user asking "what did the last alignment find?" can be answered by reading one file, no re-running the skill.
3. Warnings that persist across runs are visible as continuity (same warning text appears in successive reports) rather than opaque counts.
4. The skill remains idempotent: re-running produces the same report file content if nothing changed.

### Related

- Session 46 incident: agent could not answer "what are the two warnings about?" without re-running checks
- DSM_0.2 "Earn Your Assertions": the skill currently asserts "2 warnings" without showing the evidence
- DSM_0.2 "Know Your Context": future sessions lose context that the prior session had

## Proposal #51: Tighten Session Transcript Protocol self-check to cover pure-reasoning turns

**Source:** Session 46 incident (see Entry 58 in `2026-04-08_s46_methodology.md`)
**Priority:** High
**Scope:** DSM_0.2 Session Transcript Protocol section

### Problem statement

The Session Transcript Protocol's turn-boundary self-check rule, as currently worded in the project CLAUDE.md reinforcement block, reads (paraphrased):

> "If your first tool call this turn was not a transcript append **and the turn requires any tool calls**, the protocol was violated."

The `AND the turn requires any tool calls` clause creates a blind spot: a pure-reasoning turn (decision analysis, recommendation, trade-off evaluation, conceptual Q&A) has zero required tool calls, so the second conjunct is false and the self-check never fires. The agent produces its reasoning in chat output only, and nothing gets appended to the transcript. The violation is invisible from inside the turn.

Session 46 hit this exact failure mode: a ~60-line Sprint 16 decision analysis (five-question gate evaluation, Option A vs B, recommendation, proposed sprint shape) was produced as a pure-text turn with zero tool calls. The transcript stayed empty for that turn. The user had to notice and flag it manually.

### Why this is worse than it looks

Pure-reasoning turns contain the highest-value thinking for reasoning-efficiency analysis:
- Decision rationale
- Considered-and-rejected paths
- Risk framing and hedges
- Narrative shaping that drives downstream execution

Losing this content means the transcript preserves *mechanical* reasoning (which files to read, which tools to call) but not *judgment* reasoning. The transcript becomes a log of actions, not a log of thinking, which is the inverse of its stated purpose.

### Proposed solution

Remove the `AND the turn requires any tool calls` conjunct. Restate the rule positively:

> "Every turn begins with a transcript append. Every turn. If the first tool call of a turn is not a transcript append, the protocol was violated. This includes turns that produce reasoning, recommendations, or decisions without touching files."

If the original intent of the `AND` clause was to exempt trivial acknowledgments ("OK, proceeding"), the exemption should be content-based ("no new reasoning this turn"), not tool-call-based. Multi-paragraph decision analyses are never content-trivial.

### Acceptance criteria

1. DSM_0.2 Session Transcript Protocol section states the rule positively, without a tool-call-count condition.
2. An explicit example of the pure-reasoning-turn failure mode is included in the protocol section.
3. Agents reading the protocol can recognize that decision analyses, recommendations, and Q&A turns all require a transcript append before the response is produced.
4. The `validate-transcript-edit.sh` PreToolUse hook is unaffected (it enforces shape, not occurrence). Occurrence enforcement stays with the behavioral protocol + self-check rule, which is now tightened.

### Related

- Entry 58 (Session 46 methodology): full incident narrative and root cause analysis
- Existing `UserPromptSubmit` hook in `.claude/settings.json`: injects per-turn reminder, but does not prevent pure-reasoning turns from slipping through because the reminder is advisory, not blocking
- DSM_0.2 "Process narration" requirement: the protocol already says thinking blocks should narrate "considered-and-rejected paths, doubts, loops, and reversals", which is precisely what pure-reasoning turns contain
