## Proposal #52: PGB Concept Gate Granularity Guidance

**Target section:** DSM_0.2 §17.1 (Pre-Generation Brief Protocol)
**Priority:** Medium
**Origin:** Entry 59 (methodology)

### Problem

The PGB four-gate model (concept → implementation → run, plus the added
collaborative definition gate) does not specify gate granularity. For
complex artifacts with internal design decisions (thresholds, condensation
rules, output budgets), the concept gate can be satisfied at the artifact
level while leaving critical design decisions unreviewed.

S47 evidence: BL-302 passed its concept gate. When implementing the core
module (`knowledge_summary.py`), the agent skipped the module-level concept
gate and went from function signatures directly to TDD + code. The result
was a module with an unbounded hierarchy component that would produce
~4,700 lines for DSM Central instead of the ~200-line target. The user
caught the gap during review.

### Proposal

Add to DSM_0.2 §17.1:

> For complex artifacts (modules with multiple internal design decisions,
> algorithms with configurable thresholds, output formats with size
> constraints), the concept gate covers the artifact's internal logic,
> not just its existence. Present: what decisions the artifact makes,
> what rules govern those decisions, and what gets excluded. Function
> signatures are implementation briefs, not concept briefs.
>
> **Trigger heuristic:** If the artifact has parameters that affect what
> appears in the output (thresholds, top-N limits, depth limits, line
> budgets), those parameters and their rationale are concept-gate material.

### Expected Impact

Prevents unbounded/incorrect implementations from passing concept review
when the artifact has internal decisions that affect behavior. Catches
design gaps before code is written rather than during review.

## Proposal #53: Vision Without Speculative Implementation

**Target section:** DSM_6.0 §1.9 (Think Ahead)
**Priority:** Medium
**Origin:** Entry 60 (methodology)

### Problem

The existing "Don't design for hypothetical future requirements" principle
is implicit in DSM but not stated explicitly in §1.9 Think Ahead. When the
vision spans multiple future phases (e.g., the Intrinsic-ToC vision has
four layers), the tension between vision documentation and scope discipline
needs an explicit principle.

S47 evidence: During Sprint 16 design, the agent proposed a data/format
separation to enable future JSON and MCP output formats. The user correctly
identified this as speculative: Sprint 16 only needs markdown, no second
format is planned. Without an explicit principle, the agent was pattern-
matching on "good architecture" rather than "deliverable-scoped design."

### Proposal

Add to DSM_6.0 §1.9 (Think Ahead):

> **Vision-directed, deliverable-scoped.** Plans should articulate the
> full vision (layers, future phases). Implementation should target only
> reachable, urgent deliverables. The vision justifies the direction;
> the deliverable justifies the code. If a design choice only pays off
> in a future phase with no concrete timeline, document it in a research
> doc, do not implement it.

**Placement note:** User (S47) suggested §1.9 Think Ahead. Open to
discussion on exact placement when the BL is implemented.

### Expected Impact

Explicit principle gives agents and contributors a clear decision rule
when forward-looking architecture tempts over-engineering. Complements
existing §22 Protocol Violation Triage and the PGB concept gate.

## Proposal #54: Sprint Boundary Gate in Wrap-Up Flow

**Target:** `scripts/commands/dsm-wrap-up.md`, CLAUDE.md sprint boundary
section, optional new `/dsm-sprint-boundary` skill
**Priority:** Medium
**Origin:** Entry 61 (methodology)

### Problem

Sprint Boundary Checklist exists in CLAUDE.md (7 items: checkpoint, feedback,
decisions, blog journal, README, epoch plan, hub/portfolio notification) but
has no automatic trigger. When a sprint completes, the agent defaults to
session-level wrap-up (`/dsm-wrap-up`) without running the sprint-level
checklist first. S47 evidence: Sprint 16 Phase 1 completed, agent suggested
session wrap-up, user caught the gap.

### Proposal

Three complementary changes:

1. **`/dsm-wrap-up` pre-step:** Detect sprint boundaries by checking epoch
   plan status. If active sprint is complete, run Sprint Boundary Checklist
   before session-level tasks.

2. **CLAUDE.md gate statement:**
   > Sprint boundary gate (before any wrap-up): If this session completed a
   > sprint, run the Sprint Boundary Checklist before suggesting session
   > wrap-up. Do not conflate session-end with sprint-end.

3. **Optional `/dsm-sprint-boundary` skill:** User-invocable, walks the
   7-item checklist interactively.

### Expected Impact

Prevents informal "substantially complete" sprint states. Closes the gap
between verbal acknowledgment and procedural closure.

## Proposal #55: Verification-Triggered Checkbox Reconciliation

**Target:** `scripts/commands/dsm-go.md` Step 3.6, sprint close protocol
**Priority:** Medium
**Origin:** Entry 62 (methodology)

### Problem

Even when the sprint boundary checklist is verified (e.g., during `/dsm-go`
Step 3.6), the epoch plan's in-plan checkboxes do not get reconciled with
actual state. S47 evidence: Sprint 15 verified 7/7 complete, but the epoch
plan's checklist still shows only 1/7 boxes ticked.

### Proposal

1. **Extend `/dsm-go` Step 3.6:** After verifying each checklist item, edit
   the corresponding checkbox in the epoch plan. `[ ]` → `[x]` when verified
   complete. Flag discrepancies (e.g., `[x]` marked but verification fails).

2. **Sprint-close protocol step:** When changing sprint status from
   IN PROGRESS to COMPLETE, update every checkbox to reflect actual state.
   No box remains `[ ]` at closure.

### Expected Impact

Makes epoch plans a reliable source of truth for sprint state, not just
sprint intent. Enables future automation to flag stale unchecked boxes
in completed sprints.

## Proposal #56: Epoch Boundary Checklist

**Target:** CLAUDE.md alignment template, new `/dsm-epoch-boundary` skill
**Priority:** Medium
**Origin:** Entry 63 (methodology)

### Problem

Third boundary-protocol gap (after Entry 61/62): sprint boundaries have a
7-item checklist; epoch boundaries have nothing. `/dsm-finalize-project`
has an epoch retrospective template but runs only at PROJECT finalization.
Epochs accumulate sprint closures but never get explicit closure.

### Proposal

1. **Add Epoch Boundary Checklist to CLAUDE.md alignment template** (9 items):
   epoch retrospective, sprint boundaries verified, blog materials compiled,
   epoch plan COMPLETE, metrics summary, deferred requirements themed,
   next epoch plan drafted, hub/portfolio notified, decision log reviewed.

2. **Extract `/dsm-epoch-boundary` skill** from `/dsm-finalize-project`.
   Standalone skill that runs at each epoch close, reusing the existing
   retrospective template. `/dsm-finalize-project` calls it for any
   unclosed epochs at project end.

3. **Cross-reference with Proposals #54/55:** Sprint boundary gate detects
   when a sprint close also closes an epoch, triggers epoch boundary flow.

### Expected Impact

Parallels sprint boundary treatment at epoch scale. Captures epoch-level
learning while fresh. Standardizes the "themed deferred requirements"
pattern S47 produced ad hoc.
