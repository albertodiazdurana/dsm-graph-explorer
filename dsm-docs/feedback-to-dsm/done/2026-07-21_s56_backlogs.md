## Proposal 58: `/dsm-research-add` assumes a README sub-table structure that spokes may not have

**Problem.** `/dsm-research-add` Step 6 instructs the agent to "insert a row into
`dsm-docs/research/README.md` under the chosen sub-table heading" and specifies
five numbered sub-tables (Tied to an active BL, Informs an active BL, Ready for
promotion, Tool/methodology assessments, Untracked carryover), each with its own
row format.

In this spoke, `dsm-docs/research/README.md` has none of that structure. It has a
single `## Files` table with `File | Purpose` columns. The skill's instruction was
therefore unfollowable as written, and the agent had to choose between imposing a
structure the file does not have (a silent restructure the user did not approve)
or matching the actual format and reporting the divergence.

**Why it matters.** The failure is silent in the bad direction: an agent that
follows the skill literally will restructure a spoke's README as a side effect of
adding one research file, widening an approved edit. The Cross-Repo / approved-diff
discipline elsewhere in DSM explicitly forbids that shape of widening.

**Proposal.** Add a shape-detection step to `/dsm-research-add` Step 6: if the
target README does not contain the expected sub-table headings, degrade to
appending a row in the file's existing table format and report the divergence to
the user, rather than creating the assumed structure. Optionally, offer
restructuring as a separate confirmed action.

**Severity:** Low-Medium. Wrong output is recoverable, but it is a silent
protocol-vs-reality mismatch of the class DSM generally catches.

## Proposal 59: add pre-registration of falsification conditions to the experiment protocol

**Problem.** DSM_2.0.C and the experiment-documentation standard require
pre-registered success criteria, which S53's EXP-011 followed well. What is not
named is the inverse and arguably more valuable artifact: a pre-registered
**falsification condition for the agent's own recommendation**, written before the
measurement runs.

**Evidence from this session.** Before running a parameter sweep, the agent wrote
down that a flat metric curve (rather than a transition) would mean its own
recommended design family was dead. The curve came back flat, and the design died
cleanly on a condition that could not be renegotiated after the fact. Without the
pre-registration, a flat curve reads naturally as "no strong signal either way"
and the recommendation survives on inertia.

The same session also produced the failure this guards against: a quality metric
(NMI against directory labels) was used for most of a session as evidence of
success, and a null-model test later showed a randomized graph scored the same.
The metric could not have failed, so it never tested anything.

**Proposal.** Two additions to the experiment protocol:
1. When the agent recommends a design and then measures it, require a written
   falsification condition BEFORE the measurement.
2. Require any quality/gate metric to be validated against a null or control
   ("could this metric have come out the other way?") before it is used to
   justify a decision.

**Severity:** Medium. Both are cheap, and both caught real errors here that
would otherwise have shipped into a sprint.
