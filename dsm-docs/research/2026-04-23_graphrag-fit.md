# GraphRAG Fit Analysis for DSM

**Date:** 2026-04-23
**Session:** 49
**Author:** Alberto Diaz Durana (with 3 parallel Sonnet research agents)
**Status:** Research document, INPUT to planning pipeline (not an actionable BL)
**Scope:** Assess whether GraphRAG is worth adopting for DSM across three targets:
(a) DSM methodology corpus, (b) DSM Graph Explorer repo, (c) Ecosystem Graph / Avatar Layer 3.

---

## TL;DR

**Recommendation: adopt ideas only, do not adopt GraphRAG as a framework.**

GraphRAG's graph-topology ideas (Leiden community detection, hierarchical community patterns) are valuable and already on the Sprint 18 roadmap via BL-302 Phase 2. Its retrieval wrapper (entity-extraction via LLM, community summaries authored by LLM, vector index over summaries) directly contradicts DEC-009 ("no local LLM dependencies; GE's value is structural analysis") and the Intrinsic-ToC design position ("agent IS the query engine"). Independent benchmarks show GraphRAG loses to vanilla RAG on single-hop queries, suffers a 34% entity-miss ceiling from imperfect extraction, and carries $33K-class indexing costs that killed enterprise pilots. The 2026 production consensus converges on hybrid vector+BM25+rerank, not GraphRAG.

Minimal experiment proposed: run Leiden on GE's existing Central graph (4,703 nodes), surface top 5-10 clusters in the TOON knowledge summary, human-evaluate coherence **without** LLM-authored summaries of the clusters. This captures the structural insight from GraphRAG that is compatible with DEC-009.

## 2. Landscape Summary (from Agent A)

### 2.1 Mechanism

GraphRAG inserts a structured intermediate representation between corpus and LLM:
1. **Entity/relation extraction** via LLM passes over the corpus (expensive).
2. **Property graph construction** from extracted entities + relations.
3. **Leiden community detection** applied hierarchically at multiple granularities.
4. **Community summaries** pre-generated by LLM and stored per cluster.
5. **Query modes:** "local" (traverse from entity anchors) and "global" (aggregate across community summaries for sensemaking questions).

The community-summary layer is the structural differentiator vs vanilla RAG.

### 2.2 Leading Implementations (April 2026)

| Implementation | Indexing Cost Signal | Query Modes | Maturity | License | Community |
|---|---|---|---|---|---|
| **Microsoft GraphRAG** | $4-7 per 100K tokens; LazyGraphRAG variant cuts to <$5 for large corpora by deferring summaries | local, global | Stable, active | MIT | ~30K stars |
| **LightRAG** (HKUDS, EMNLP 2025) | ~$0.15 for corpus costing $4-7 on MS GraphRAG (vendor claim) | local, global, hybrid, naive | Approaching stable | MIT | ~28K stars |
| **HippoRAG 2** (OSU NLP, ICML 2025) | 10-30x cheaper multi-hop than MS GraphRAG (paper) | local/associative (Personalized PageRank, no community summaries) | Research code | Apache 2.0 | Smaller |
| **Neo4j GraphRAG Python** | Depends on assembly | vector, graph, hybrid | Stable, vendor-backed | Apache 2.0 | Neo4j ecosystem |
| **LlamaIndex PropertyGraphIndex** | Framework, not opinionated | pluggable | Stable | MIT | ~35K stars (LlamaIndex) |

Notable in flux: PathRAG, HyperGraphRAG, LinearRAG (arxiv-stage). **GraphRAG-Bench (ICLR 2026)** is the first rigorous standardized evaluation suite; results are emerging.

### 2.3 Documented Wins

- **Global sensemaking over large corpora.** Edge et al. (2024) showed GraphRAG's global mode beats vector RAG on "main themes across corpus" queries.
- **Multi-hop reasoning under entity-dense queries.** arxiv 2604.09666 confirms GraphRAG's stability advantage; ~86% vs ~32% accuracy on multi-hop tasks (secondary citation, unverified).
- **Corpus-spanning queries with 10+ entities.** Vector RAG degrades; GraphRAG sustains (secondary citation).
- **Stability in agentic search pipelines** when indexing cost is amortized across many queries.

**Honest caveat:** Most benchmarks use LLM-as-judge, which carries position and vendor biases.

## 3. Critiques Summary (from Agent B)

### 3.1 Failure Modes (with numbers)

**arxiv 2502.11371 ("RAG vs GraphRAG: A Systematic Evaluation", Feb 2025):**
- NQ single-hop: vanilla RAG F1 **68.18%** vs Community-GraphRAG **65.44%**. Vanilla wins.
- NQ KG retrieval accuracy: GraphRAG **39.2%** vs standard RAG **86.7%**.
- **Only 65.5% of answer entities appear in the constructed KG.** The 34% entity-miss rate is a structural ceiling, not a tunable parameter.
- HotpotQA multi-hop: GraphRAG 64.60% vs RAG 63.88%. Marginal GraphRAG win. Hybrid strategy wins clearly (77.62%).

**Production failure classes** (2026 practitioner literature):
- Entity resolution errors compound across hops; below 85% resolution accuracy, multi-hop paths produce junk.
- Over-retrieval loops when confidence calibration breaks.
- Hallucination on "null" queries: global search fabricates rather than returning "insufficient information."
- LLM-as-judge position bias can flip benchmark sign based on output ordering.

### 3.2 Cost Critique

- **Indexing token burn:** "Several to dozens of times more tokens than the original text" (RAGFlow 2025 review). $120 reported for 1,000-page PDF using GPT-4-Turbo.
- **$33K indexing cost** for large-scale datasets on Microsoft GraphRAG (Graph Praxis, 2026) — widely cited as the number that killed enterprise pilots.
- **Construction time:** 5,560s vs 135s for RAG on same corpus (41x slower, arxiv 2502.11371).
- **Re-indexing:** proportional to impacted graph portion, not just the changed document. High-update-rate corpora compound cost.
- **Query-time cost:** ~$0.40/query with GPT-4-Turbo (GitHub microsoft/graphrag #440).

### 3.3 Head-to-Head Benchmarks

- **GraphRAG vs hybrid BM25+dense+rerank:** No direct benchmark found. Hybrid yields 15-25% accuracy gains over single-retrieval with no graph infra needed (RAGFlow 2025).
- **GraphRAG vs long-context (Gemini 2M, Claude 1M):** No dedicated benchmark. Long-context has ~30% "lost in the middle" degradation; $0.10/query vs $0.00008 for RAG (1,250x gap). Long-context wins for <100 docs / <100K tokens.

### 3.4 Alternative Approaches

- **Agentic RAG** — LLM plans iterative retrieval; wins when queries are diverse/unpredictable.
- **Long-context eats RAG** — partially playing out for small corpora; cost/latency prevents wholesale replacement at enterprise scale.
- **Structured-document retrieval (TOON, YAML front-matter, XML, "README for LLMs")** — wins when knowledge structure is maintained by humans; avoids entity-resolution noise entirely. *This is the DSM direction.*
- **MCP-style tool retrieval** — wins when data is already in queryable structured form.
- **LazyGraphRAG / hybrid-lite** — community responses to cost critique.

### 3.5 2026 Verdict

**Maturing into niche use**, not becoming mainstream. Specialist tool for large, relationship-dense, **stable** corpora (biomedical, legal networks, supply-chain ontologies). Production default converging on hybrid vector+BM25+rerank.

**Vendor-bias flag:** FalkorDB publishes GraphRAG cost-reduction guides; they have direct interest in adoption.

## 4. Fit Analysis per Target (from Agent C)

### 4.1 Target (a) — DSM Methodology Corpus

**Verdict: complement in principle, premise mismatch in practice**

The DSM Central corpus (hundreds of markdown files: protocols, decisions, backlogs, transcripts) is exactly the corpus type GraphRAG was designed for. However, the Intrinsic-ToC vision (`dsm-docs/research/2026-04-13_intrinsic-toc-vision.md` §3) has staked out a conflicting design position: **the agent IS the query engine**. Intrinsic-ToC is a pre-computed static artifact read directly by the agent. GraphRAG requires an embedding + retrieval layer between corpus and agent. DSM has chosen knowledge compilation over retrieval (§7, Karpathy pattern).

**Counter-evidence (v1.6.0 BL-385):** At portfolio scale (thousands of files across many repos), static-ToC threshold-cutting may lose information. GraphRAG's hierarchical community model handles scale more gracefully. The premise mismatch may matter less than the scale-handling gap.

### 4.2 Target (b) — DSM Graph Explorer

**Verdict: contradiction with DEC-009**

GE already builds a graph (4,703 nodes, 5,041 edges on Central). What GraphRAG *adds* is:
- Embed nodes (requires sentence-transformer or API calls per node) — violates DEC-009.
- LLM-author community summaries (the exact redundancy DEC-009 rejects) — violates DEC-009.
- Vector-search infrastructure — new dependency, violates DEC-009 §Rationale.

DEC-009 is not a cost tradeoff; it is a design position that GE's value is **deterministic structural analysis**. GraphRAG replaces deterministic computation with probabilistic LLM summarization. These are architecturally opposed. DEC-010 (TOON migration) reinforces the direction toward token-efficient static output, not a dynamic retrieval layer.

**Counter-evidence:** Leiden community detection is **separable** from GraphRAG's embedding/retrieval wrapper. Sprint 18 already plans Leiden clustering via BL-302 Phase 2 (`dsm-docs/plans/epoch-5-plan.md` §Sprint 18). The structural clustering algorithm is on-roadmap; only the retrieval wrapper conflicts with DEC-009.

### 4.3 Target (c) — Ecosystem Graph / Avatar Layer 3 (Sprint 20)

**Verdict: informative complement at the pattern level; do not adopt the implementation**

Sprint 20 targets cross-repo references, ecosystem-level graph connecting spoke ToCs, and a persistence-strategy decision. This is structurally similar to GraphRAG's "global" query tier. The community-summarization pattern is directly relevant as a design reference: it answers the open question in `intrinsic-toc-vision.md` §9.5 ("Avatar materialization: FalkorDB? linked Intrinsic-ToCs?") with "hierarchical community summaries stored as a graph."

The mechanism diverges: GraphRAG materializes summaries via LLM authoring; Layer 3's design position is that cross-repo navigation is done by the consuming agent reading linked static ToCs.

**Counter-evidence:** At 5-10 repos, linked Intrinsic-ToCs may be entirely sufficient and GraphRAG's community hierarchy is over-engineered. The ecosystem graph at Sprint 20 scale may not stress-test the static-ToC model enough to reveal its limits.

## 5. Cross-Cutting Synthesis

Convergence across the three reports:

1. **Cost is decisive.** Agent B's $33K indexing figure and 41x construction time gap are the single strongest case against adoption. Not a tunable knob; an architectural cost center.
2. **Entity-miss ceiling is structural.** 34% of answer entities missing from automatically extracted KGs (arxiv 2502.11371) means GraphRAG cannot exceed that ceiling without human-in-the-loop KG curation. DSM corpora (protocols, decisions, session transcripts) are entity-sparse relative to biomedical literature; the ceiling would likely be worse for DSM, not better.
3. **GraphRAG's best idea is already on roadmap.** Sprint 18 BL-302 Phase 2 plans Leiden clustering. The graph-topology component that is actually valuable is being adopted without the retrieval wrapper.
4. **DSM's trajectory is structured-doc retrieval (Agent B §3.4), not GraphRAG.** TOON migration (DEC-010) + Intrinsic-ToC ("README for LLMs") are explicit alternatives to retrieval-layer approaches. They are the same category Agent B identifies as "wins when knowledge structure is maintained by humans."
5. **Retrieval-wrapper variants (LightRAG, HippoRAG 2) still assume local/hosted LLM summarization.** LazyGraphRAG defers but does not eliminate. None escape the DEC-009 conflict.

## 6. Recommendation

**Adopt ideas only.**

- **Leiden clustering:** Already on Sprint 18 roadmap via BL-302 Phase 2. No new action needed; confirm during Sprint 18 kickoff that the implementation stays on the "structural clustering without LLM summaries" side of the line.
- **Community-hierarchy pattern:** Worth studying as a design reference when Sprint 20 Avatar Layer 3 design begins. The pattern informs, the mechanism does not transfer.
- **Do not adopt:** Microsoft GraphRAG, LightRAG, HippoRAG 2, or Neo4j GraphRAG as frameworks. All require the embedding + LLM-summarization stack that violates DEC-009.
- **Do not reverse DEC-009.** The research surfaced no concrete DSM demand that would justify introducing local LLM or embedding dependencies. DEC-009 §Consequences leaves the door open "given a concrete problem statement"; no such statement has emerged.

## 7. Minimal Experiment Proposal

**Goal:** Validate that graph topology alone (no LLM-authored summaries) produces coherent community structure for DSM Central.

**Steps:**
1. Run Leiden on GE's existing 4,703-node Central graph (Sprint 18 BL-302 Phase 2 scope).
2. Extract top 5-10 clusters.
3. Surface cluster membership in the TOON knowledge summary (DEC-010 output format).
4. Human evaluation: are clusters semantically coherent **without** LLM-authored summaries? (Author + 1 reviewer, simple rubric: coherent / partially coherent / incoherent.)

**What this tests:** whether structural signal alone is strong enough to support Layer 3 pattern use, or whether LLM summarization would be required to make clusters useful. If structural signal is sufficient, DEC-009 is upheld in practice. If not, this becomes the "concrete problem statement" DEC-009 §Consequences anticipates.

**Scope:** Sprint 18 already plans steps 1-2; this proposal adds step 3 (TOON surfacing) and step 4 (evaluation) as low-cost extensions. Not a new BL; an amendment to BL-302 Phase 2 scope to include a coherence check.

## 8. Open Questions / Flagged Assumptions

1. **Central corpus scale trajectory.** Fit analysis for target (a) assumed hundreds of files. If Central grows to thousands, static-ToC scaling becomes the binding constraint. Revisit if portfolio footprint doubles.
2. **Sprint 20 has no detailed plan file yet.** Target (c) fit assessment uses research-document design intent as proxy. Revisit during Sprint 20 kickoff.
3. **GraphRAG-as-service vs GraphRAG-as-local-deps.** DEC-009 targets local dependencies. An API-only GraphRAG (no local model) is not explicitly ruled out. Analysis treats the pre-computed-vs-retrieval philosophical conflict as the harder constraint; this is an interpretive judgment.
4. **LLM-as-judge bias.** Most GraphRAG wins in the literature use LLM-as-judge evaluation. GraphRAG-Bench (ICLR 2026) is emerging but not yet widely replicated. Cited numbers should be treated as directional.
5. **FalkorDB vendor bias.** DSM uses `falkordblite` as optional graph dep. FalkorDB publishes GraphRAG cost-reduction guides. Check for vendor pull if a future BL proposes GraphRAG adoption.
